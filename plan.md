### 第一阶段：数据库与领域模型设计 (The Foundation)

**设计意图：** 这一步决定了系统的地基。必须体现报告中提到的“乐观锁” 和“全生命周期”  的数据支撑。

**📋 投喂给 AI 的提示词（Copy 这段）：**

> **角色：** 资深数据库架构师
> **背景：** 基于 Spring Boot 开发《企业固定资产全生命周期管理系统》。
> **任务：** 设计 MySQL 数据库的详细 ER 结构，并输出带注释的 DDL SQL 语句。
> **核心表结构要求（必须严格遵守）：**
> 1. **系统基础表 (RBAC)：**
> * `sys_user` (用户), `sys_role` (角色), `sys_menu` (权限), `sys_user_role`, `sys_role_menu`。
> * *注意：* 必须包含 `dept_id` (部门ID) 以支持报告中的“跨部门调拨”场景。
> 
> 
> 2. **资产主表 (`ast_asset`) - 核心中的核心：**
> * **唯一标识：** 必须包含 `uuid` 字段（全局唯一标识符）。
> * **并发控制：** 必须包含 `version` 字段（用于 MyBatis-Plus 乐观锁插件）。
> 
> 
> * **价值字段：** `original_value` (原值), `net_value` (净值), `residual_rate` (残值率)。
> * **状态字段：** `status` (枚举值：闲置、在用、维修、报废) 。
> 
> 
> * **分类外键：** `category_id` (关联分类表)。
> 
> 
> 3. **全生命周期记录表：**
> * `ast_operation_record`: 记录资产的所有变动（入库、领用、退库、维修、报废）。记录 `operator_id` (操作人), `operate_time`, `operate_type`, `description`。
> 
> 
> 4. **折旧记录表 (`ast_depreciation`)：**
> * 记录每次自动折旧的金额、日期，用于财务审计 。
> 
> 
> 
> 
> **通用规范：**
> * 所有表必须包含 `is_deleted` (逻辑删除)、`create_time`、`update_time`。
> * 字段注释必须详细，能够直接用于生成毕业论文的数据库文档。

### 第三阶段：核心业务功能实现 (The Core Logic)

**设计意图：** 这是论文最“加分”的部分。重点攻克“二维码” 、“折旧算法” 和“闲置盘活” 。

**📋 投喂给 AI 的提示词（Copy 这段）：**

> **角色：** 全栈高级开发工程师
> **任务：** 实现《资产全生命周期》的核心业务模块。
> **模块一：资产入库与二维码 (Asset Entry & QR)**
> * **逻辑：** 新增资产时，自动生成 UUID。
> * **工具：** 集成 Google `ZXing` 库。
> * **代码要求：** 编写一个工具类方法，传入 UUID，生成二维码图片的 Base64 字符串，并在查询详情接口中返回该字符串 。
> 
> 
> 
> 
> **模块二：自动折旧引擎 (Depreciation Engine)**
> * **逻辑：** 依据开题报告，系统需内置“平均年限法”折旧算法 。
> 
> 
> * **实现：** 使用 `@Scheduled(cron = "0 0 0 1 * ?")` 创建定时任务，每月 1 号凌晨遍历所有状态为“在用”的资产，计算本月折旧额，更新 `net_value` (净值)，并插入一条折旧记录。
> 
> 
> **模块三：资产流转与状态机 (Workflow)**
> * **逻辑：** 实现资产的 领用 (Apply) -> 退库 (Return) -> 维修 (Fix) -> 报废 (Scrap)。
> * **约束：** 必须校验状态流转的合法性（例如：只有“闲置”的资产才能被“领用”；“报废”资产必须停止折旧）。
> 
> 
> 
> 
> **模块四：闲置资产盘活 (Idle Asset Activation)**
> * **逻辑：** 编写一个查询接口，专门筛选 `status = 'IDLE'` 且 `update_time` 超过 30 天的资产，用于在前端“闲置公示区”展示，支持跨部门申请调拨 。
>
